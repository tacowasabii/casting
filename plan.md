# 캐스팅 ATS 제품 기획안 (가칭: 캐스트보드)

## 1. 제품 정의

영화·드라마 캐스팅 디렉터를 위한 지원자 관리 시스템(Applicant Tracking System).
공고 게시 이후의 업무 — 지원 수집, 선별, 감독 컨펌, 오디션 안내 — 를 하나의
워크플로우로 통합한다. 배우 회원가입이 필요 없는 단면(single-sided) 도구로,
필름메이커스·인스타그램 등 기존 공고 채널과 경쟁하지 않고 그 위에서 동작한다.

**한 줄 정의**: "이메일과 엑셀과 카톡에 흩어진 캐스팅 업무를 한 화면으로."

**핵심 가설**: 캐스팅 디렉터의 업무 시간 중 배우를 '찾는' 시간보다 지원자를
'정리하고 컨펌받는' 시간이 더 크며, 이 정리·컨펌 구간을 자동화하면 유료 결제
의사가 발생한다.

## 2. 타겟 사용자

- **주 사용자 (결제자)**: 영화·드라마 캐스팅 디렉터. 1~3인 팀, 작품 단위 계약,
  동시 1~3개 작품 진행. 조단역 배역 30~80개를 공고 기반으로 캐스팅하며,
  배역당 수십~수백 건의 이메일 지원을 수작업으로 처리하는 사람.
- **부 사용자 (비로그인)**: 감독·연출부. 디렉터가 보낸 리뷰 링크를 카톡으로
  받아 모바일에서 후보를 검토하는 사람. 회원가입 없음.
- **간접 사용자 (비회원)**: 지원 배우. 지원 폼 또는 이메일로 지원하며,
  서비스의 존재를 인지할 필요 없음. 오디션 확정 시 알림톡/문자를 수신.
- **초기 검증 대상**: 지인 캐스팅 디렉터 1인 (영화·드라마, 주연~단역 전 범위
  담당). 검증 기준은 "다음 프로젝트에서 요청 없이 재사용하는가."

## 4. 모듈별 상세 기획

### 4.1 프로젝트 / 배역 관리

- **구조**: 작품(Project) > 배역(Role) 2계층.
- **작품 생성 필드**: 작품명, 유형(영화/드라마/웹드라마/기타), 제작사,
  촬영 기간, 메모.
- **배역 생성 필드**: 배역명, 모집 인원, 성별, 나이대(범위), 키(범위, 선택),
  상세 조건(자유 텍스트: 사투리, 특기, 노출 여부 등), 촬영 예정일,
  지원 마감일, 출연료(선택, 내부 메모).
- **배역 상태 머신**: `모집중 → 검토중 → 감독컨펌 → 오디션 → 확정 → 종료`.
  상태는 수동 변경이 기본이되, 이벤트 발생 시 자동 전환 제안
  (예: 리뷰 링크 생성 시 "감독컨펌으로 변경할까요?").
- **대시보드 화면**: 작품별로 배역 카드가 상태 컬럼(칸반)으로 배치.
  각 카드에 신규 지원자 수 뱃지, 마감 D-day, 감독 응답 대기 표시.
  로그인 직후 첫 화면이며 제품의 인상을 결정한다.

### 4.2 지원자 수집 (인박스)

배역 생성 시 자동으로 두 채널이 발급된다.

- **① 지원 폼 (v1 우선)**: `apply.서비스명.com/r/{roleId}` 형태의 공개 링크.
  필드 — 이름, 생년, 키, 연락처, 소속사(선택), 프로필 파일(PDF/이미지,
  최대 3개), 연기 영상(파일 업로드 또는 유튜브·드라이브 링크),
  자기소개(선택). 모바일 제출 최적화 필수. 제출 완료 화면에
  "지원이 접수되었습니다" + 배역·작품명 표기.
- **② 전용 이메일 (v1.5)**: `{roleSlug}@in.서비스명.com`.
  SendGrid/Postmark Inbound Parse로 수신 → 발신자·제목·본문·첨부 추출 →
  지원자 카드 자동 생성. 파싱 실패분은 "미분류 인박스"에 쌓아 디렉터가
  드래그로 배역에 배정. 완벽한 자동 파싱을 목표하지 않고 '분류 대기함'
  UX로 엣지케이스를 흡수한다.
- **중복 판정**: 폼은 전화번호, 이메일은 발신 주소 기준. 동일 인물 재지원 시
  기존 배우 레코드에 지원 이력 추가.

### 4.3 선별 보드

배역 클릭 시 진입하는 지원자 검토 화면. **처리 속도가 제품의 핵심 가치.**

- **레이아웃**: 좌측 지원자 리스트(썸네일 + 이름·나이·키), 우측 상세 패널
  (프로필 이미지/PDF 뷰어, 영상 인라인 플레이어, 지원 정보, 메모).
- **분류 체계**: 관심 / 보류 / 제외 3단계. 키보드 단축키 — `↑↓` 이동,
  `1/2/3` 분류, `Space` 영상 재생, `M` 메모. 마우스 없이 지원자 100명을
  훑을 수 있어야 한다.
- **필터·정렬**: 나이, 키, 성별, 소속사 유무, 분류 상태, 지원일.
- **영상 처리**: 업로드 영상은 Mux(또는 Cloudflare Stream)로 트랜스코딩,
  유튜브·드라이브 링크는 임베드. 썸네일 자동 추출.
- **목표 성능 지표**: 지원자 1인당 검토 체류 시간 15초 이내에 다음 카드로
  넘어갈 수 있는 인터랙션.

### 4.4 리뷰 링크 (감독 공유)

- **생성 플로우**: 선별 보드에서 '관심' 후보 다중 선택 → "숏리스트 만들기" →
  링크 제목(예: "형사2역 1차 후보"), 마감일, 코멘트 허용 여부 설정 →
  링크 생성 → 카톡 공유 버튼.
- **감독 측 화면 (비로그인, 토큰 URL)**:
  - 카톡 OG 미리보기: "○○(작품명) · 형사2역 후보 8명 · ○일까지"
  - 랜딩: 작품·배역·마감 안내, "검토 시작" 버튼
  - 카드 뷰: 풀스크린 1인 1카드. 사진 → 스크롤 시 영상 인라인 재생 →
    기본 정보. 하단 고정 버튼 좋아요/보류/제외, 진행 표시 "3/8". 스와이프 지원.
  - 완료 화면: 선택 요약, "의견 전송" 버튼.
- **디렉터 측 결과 화면**: 응답 실시간 반영(좋아요 5·보류 2·제외 1),
  배우별 감독 영상 시청 시간 로깅, 미응답 시 마감 24시간 전 자동 리마인드
  (알림톡/문자, v1은 수동 재전송 버튼으로 대체 가능).
- **보안**: 추측 불가능한 토큰 URL, 만료일 설정, 열람 로그.

### 4.5 배우 라이브러리

모든 지원 이력이 배우 단위로 누적되는 개인 DB.

- **배우 프로필 페이지**: 기본 정보, 지원 이력(어느 작품 어느 배역, 당시 분류
  결과), 누적 영상·프로필 파일, 디렉터 메모, 태그(자유 태그: "부산사투리",
  "액션", "아역출신" 등).
- **검색**: 이름, 나이 범위, 키 범위, 태그, 과거 분류 결과("이전에
  '관심'이었던"), 자유 텍스트.
- **신규 배역 연결**: 배역 생성 시 조건 매칭되는 라이브러리 배우를
  "후보 제안"으로 표시 → 원클릭으로 숏리스트에 추가. 공고 없는 캐스팅의
  시작점이며, 사용 기간이 길수록 전환 비용이 커지는 리텐션 장치.

### 4.6 오디션 안내 (경량)

확정 후보에게 오디션 일시·장소 안내 발송(v1: 문자 발송용 텍스트 자동 생성 +
복사 버튼 / v1.5: 솔라피 연동 알림톡 자동 발송 + 참석 여부 응답 링크).
응답 현황을 배역 화면에 표시. 캘린더 연동은 범위 외.

## 5. 데이터 모델

```
User (디렉터 계정)
Team (1인이어도 팀 단위, 향후 협업 대비)
Project: id, team_id, title, type, production, shoot_period, status
Role: id, project_id, name, headcount, gender, age_min/max,
      height_min/max, requirements, deadline, shoot_date, status
Actor: id, team_id, name, birth_year, height, phone, email,
       agency, tags[], note   ← 팀 스코프 (디렉터의 사유 DB)
Application: id, role_id, actor_id, source(form/email/manual),
             files[], video_urls[], intro, triage(관심/보류/제외),
             note, created_at
Shortlist: id, role_id, title, token, deadline, allow_comment
ShortlistItem: shortlist_id, application_id, order
Review: shortlist_item_id, verdict(like/hold/pass), comment,
        watch_seconds, reviewed_at
```

설계 원칙:

1. **Actor와 Application의 분리** — 동일 배우의 반복 지원이 하나의 배우
   레코드로 귀속되어야 라이브러리가 성립한다.
2. **Actor의 팀 스코프** — 배우 데이터는 전역 공유가 아니라 디렉터 개인
   자산이다. 개인정보 관점에서도 이 구조가 안전하며, 지원 폼에 개인정보
   수집·이용 동의 항목 필수 포함.

## 6. 기술 스택

- 프론트+백엔드: Next.js (App Router)
- DB·인증·스토리지: Supabase
- UI: Tailwind + shadcn/ui
- 영상: Mux 또는 Cloudflare Stream
- 이메일 수신: Postmark Inbound (v1.5)
- 알림톡: 솔라피 (v1.5)
- 배포: Vercel
- 결제: 파일럿 무료 → 첫 과금 계좌이체 → 토스페이먼츠는 유료 5팀 이후

## 7. 개발 로드맵 (솔로, 12주)

- **Phase 1 — 뼈대 (1~3주)**: 인증, 팀/작품/배역 CRUD, 대시보드 칸반,
  지원 폼(파일 업로드 포함), 폼 제출 → 지원자 카드 생성.
- **Phase 2 — 선별 (4~6주)**: 선별 보드 전체(리스트+상세 패널), 키보드
  단축키, Mux 연동, PDF 뷰어, 필터.
- **Phase 3 — 리뷰 링크 (7~8주)**: 숏리스트 생성, 감독 모바일 카드 뷰,
  응답 수집·결과 대시보드, OG 미리보기.
  **← 8주차: 시연 가능 시점. 폼→선별→컨펌 완결 플로우 동작.**
- **Phase 4 — 라이브러리 + 인박스 (9~10주)**: 배우 레코드 통합·검색·태그,
  이메일 인바운드 파싱, 미분류 인박스.
- **Phase 5 — 완성도 (11~12주)**: 오디션 안내, 리마인드, 데모 데이터 세팅
  (실감나는 가상 배우 15인 — 자연스러운 이름·필모·영상), 온보딩 플로우.

## 8. 검증 계획

- **8주차**: 지인 디렉터에게 데모 시연(대면 불가 시 데모 계정 + 5분 화면
  녹화 영상 전달). 확인할 것 — 어느 화면에서 "이거 좋네"가 나오는가,
  어느 기능을 "우린 이렇게 안 하는데"라고 하는가.
- **10~14주차**: 실제 진행 중인 배역 1개에 파일럿 적용. 지원 폼 링크를
  실공고에 부착, 실지원자 수집, 실감독에게 리뷰 링크 발송까지 1사이클 완주.
- **성공 기준**: ① 디렉터가 다음 배역에서 요청 없이 재사용 ② 감독이 리뷰
  링크를 마감 내 완료 ③ "월 얼마면 쓰겠냐"에 구체적 금액 응답.
- **실패 신호**: 파일럿 후 재사용이 없거나, 결국 엑셀·카톡으로 회귀.
- **확장 검증**: 파일럿 성공 시 지인에게 동료 디렉터 2~3인 소개 요청 →
  동일 페인 여부 확인(1인 특이 케이스 리스크 제거).

## 9. 수익 모델 (가설)

- 작품당 과금 기본: 작품 1개 활성화당 월 5~9만 원(디렉터의 작품 단위 수입
  구조에 부합).
- 대안: 월 구독(작품 3개까지 7만 원선).
- 배우 측 과금 없음 — 지원 마찰을 만들지 않는 것이 도구 가치의 전제.
- 파일럿~초기 5팀은 무료 또는 명목가, 가격 검증은 "실사용 후 지불 의사"
  인터뷰로 진행.

## 10. 범위 외 (v2 이후 백로그)

계약·정산 관리, AI 프로필 자동 태깅·추천(라이브러리 데이터 축적 후),
캘린더 연동, 팀 협업(권한·코멘트), 광고·숏폼 시장용 변형, 배우 대상 기능
일체(양면 확장은 유료 10팀 이후 판단).

## 11. 리스크와 대응

- **워크플로우 가정 오류** — 현재 기획은 전부 추정 기반. 8주차 시연에서
  지인 피드백으로 보정하되, Phase 1~3은 어떤 워크플로우든 공통인 구조
  (지원 수집→선별→공유)라 폐기 리스크가 낮은 순서로 배치.
- **감독이 링크를 안 누름** — 카톡 열람 경험에 완성도 집중, 리마인드 보완,
  그래도 안 되면 디렉터가 감독 옆에서 같이 보는 '대면 모드'로 피벗.
- **시장 협소** — 영화·드라마는 검증 시장, 확장 본진은 광고·숏폼 포함 재판단.
- **개인정보** — 지원 폼 동의 절차, 배역 종료 후 보관 기간 정책,
  팀 스코프 격리로 대응.
